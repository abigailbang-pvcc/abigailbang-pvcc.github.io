    Chapter 5: Programming Algorithms and Software Testing


    Phases of computer program development:

    • Phase 1: Understand the problem

    • Phase 2: Design an algorithm to solve the problem

    • Phase 3: Write the program code in an appropriate language based on the algorithm

    • Phase 4: Enter the program and save

    • Phase 5: Execute the program

    • Phase 6: Test the program, find errors and debug, then retest

    • Phase 7: Document and evaluate the program


    Programming structures

    • Sequential structure: Top-down execution

    • Decision structures: if/then/else

    • Iterative (looping):
      • while loop: continue looping as long something is true
      • repeat/until loop: continue looping until something is true
      • for loop: loop a given number of times

    • Recursive structures:
      • Repeat a set of instructions as a subtask of itself
      • Repitition continues until:
      • degenetative case, which is a termination condition, is met (degenerative case is also called a "base case")

    Two classic programming/database problems

    • Sorting: Descending order/Ascending order
      • Sorting numbers
      • Sorting strings (in accordance with ASCII values

    • Searching: find specific data that meet a certain criteria
      • In database:
      • Filters: temporary search
      • Queries: search criteria are saved so they don't have to be recreated

    • Two popular search techniques:
      • Sequential search: search from top to bottom or bottom to top, one item at a time
      • Binary search:
      • only works if the data is SORTED first
      • start in the middle, then go to half of that, etc.

    Two classic program goals:

    • Efficiency : Program takes minimal time and/or resources
    • Correctness: Program runs in expected ways and works for all expected conditions




Software testing and verification

    • Kinds of tests:
      • Acceptance testing - Confirming whether the entire system is properly functioning
      • Integration testing - Verifying that different components of software or functions are operating together correctly
      • Unit testing - Ensuring that all software units are operating. Units are the smallest testable application component
      • Functional testing - Functions are checked by mimicking buisness scenarios that are based on functional requirements. Black box testing is a common method
      • Performance testing - Verifying how software performs with various workloads. Long testing can be used for real-life load condition performance evaluation.
      • Regression testing - New features are checked to determine whether they will break or fully degrade. Sanity testing is used to verify menus, functions and commands at the surface level if there is no time to perform a full regression test
      • Stress testing - Stress testing evaluates the strain a system is capable of managing successfully, prior to breaking. It is non-functional testing
      • Usability testing - How operable is the system or web appication for the customer to use

    • Software testing is done to detect:
      • a bug or defect can degrade interconnected systems or cause serious malfunctions.
      • companies can save millions per year in development and support if they have a good testing technique
        • Architectural flaws
        • Poor design decisions
        • Invalid or incorrect functionality
        • Security vulnerabilities
        • Scalability issues

    • Testing best practices
      • Continuous testing - Each build is tested when it becomes available. It relies on automation that is integrated with the deployment process. Due to the ability to test software in realistic environments early on, it improves the design
      • Configuration management - Allows teams to centrally maintain and track what should be tested. Relevant assets are able to be accessed, and authentication and audit trails are available with good systems to maintain compliance requirements
      • Service virtualization - Simulates services and systems which are incomplete or missing, in order to allow for earlier testing
      • Defect or bug tracking - Automated resources are utilized to monitor defects, and determine their impact, as well as related issues
      • Metrics and reporting - Reporting and analytics allow goeals statuses, and results to be shared among team members, and tools merge metrics and results in a dashboard. The health of a project can be viewed, and relationships between multiple elements can be observed
